# CLAUDE.md

Guidance for AI coding assistants working in this repository.

## About this repo

`lintr` provides static code analysis for R. It checks code for adherence to a
style guide (default: the tidyverse style guide), flags likely syntax errors,
and surfaces possible semantic problems, then reports them as `Lint` objects.
It is complementary to `styler`, which reformats code automatically.

This checkout is `d-morrison/lintr`, a fork of the upstream `r-lib/lintr`.
Upstream is the project of record. Follow upstream conventions: the tidyverse
[style guide](https://style.tidyverse.org), the tidyverse
[code review principles](https://code-review.tidyverse.org/), and
`.github/CONTRIBUTING.md`. Do not impose conventions from unrelated projects
that upstream does not follow.

## How linting works

The core pipeline:

1. `get_source_expressions()` parses a file with `parse()` +
   `utils::getParseData()`, then converts the parse data to an XML tree with
   `xmlparsedata::xml_parse_data()`. The result exposes top-level expressions
   individually (for caching) and the whole file together.
2. Each linter is a function factory. Calling it (e.g. `assignment_linter()`)
   returns a closure tagged by `Linter()` (defined in `R/utils.R`). The closure
   runs against a source expression and returns a list of `Lint` objects.
3. Most linters express their logic as an XPath query over the parse-tree XML
   (`xml2` implements XPath 1.0), then pass matching nodes to
   `xml_nodes_to_lints()` to build the lints.
4. `lint()`, `lint_dir()`, and `lint_package()` drive linters over files and
   collect the results.

Read `vignette("creating_linters")` (`vignettes/creating_linters.Rmd`) before
writing or substantially changing a linter. `pipe_call_linter` is the canonical
simple example.

## Repo layout

- `R/` holds one file per linter (e.g. `R/assignment_linter.R`) plus
  infrastructure: `get_source_expressions.R`, `lint.R`, `exclude.R`,
  `settings.R`, `cache.R`, `xml_utils.R`, `xp_utils.R`, `utils.R` (the `Linter`
  factory), `make_linter_from_xpath.R`, `make_linter_from_regex.R`,
  `linter_tags.R`, and `zzz.R` (defines `default_linters`). `DESCRIPTION`
  `Collate:` lists the load order.
- `tests/testthat/` holds the test suite. Test file names mirror the linter:
  `test-<linter_name>.R`. Snapshots live in `tests/testthat/_snaps/`. Dummy
  fixtures live in `dummy_packages/`, `dummy_projects/`, `knitr_*`, etc.
- `man/` is generated by roxygen2. Do not edit by hand.
- `vignettes/` includes `lintr.Rmd`, `creating_linters.Rmd`,
  `continuous-integration.Rmd`, and `editors.Rmd`.
- `inst/lintr/linters.csv` maps each linter to its tags. `inst/WORDLIST` is the
  spell-check allowlist. `inst/example/bad.R` is the README demo file.
- `.dev/` holds developer-only scripts (fuzz testing, branch comparison, revdep
  tooling, and the meta-tests run by the `repo-meta-tests` workflow at
  `.github/workflows/repo-meta-tests.yaml`). These are not part of the installed
  package.
- `.github/workflows/` defines CI. `.lintr` is lintr's own config for
  self-linting.

## Adding a new linter

1. Create `R/<name>_linter.R`. Suffix the function with `_linter`. The function
   is a factory returning `Linter(linter_level = "expression", function(source_expression) {...})`.
   Prefer `linter_level = "expression"` for speed; use `"file"` only when the
   logic needs the whole file. Cache static XPath strings in the factory body,
   not inside the closure.
2. Document with roxygen2 markdown. Include `@evalRd rd_tags("<name>_linter")`,
   `@seealso [linters]`, and `@export`.
3. Add the linter and its tags to `inst/lintr/linters.csv`. The
   `repo-meta-tests` CI job checks that metadata, roxygen, and the namespace
   stay in sync.
4. If the linter implements part of the tidyverse style guide, add it to
   `default_linters` in `R/zzz.R` and add a case to
   `tests/testthat/default_linter_testcode.R`.
5. Add `tests/testthat/test-<name>_linter.R`.
6. Run `devtools::document()` to regenerate `man/` and `NAMESPACE`.
7. Add a `NEWS.md` bullet for any user-facing change (tidyverse NEWS style,
   written for users).

Helpers worth using: `get_r_string()` (read a string literal as R sees it),
`source_expression$xml_find_function_calls()` (query function calls
efficiently), `make_linter_from_xpath()` / `make_linter_from_function_xpath()`
(for static XPath + static message linters), and `is_lint_level()`.

## Build, test, lint, document

Run from R with the package loaded via `devtools::load_all()`:

- `devtools::test()` runs the full test suite. Use the `filter` argument to run
  one file: `devtools::test(filter = "assignment_linter")`.
- `devtools::document()` regenerates `man/` and `NAMESPACE` from roxygen2.
- `devtools::check()` runs R CMD check. `devtools::install_dev_deps()` installs
  development dependencies first.
- `lintr::lint_package()` self-lints using the repo's `.lintr` config. CI runs
  this with `LINTR_ERROR_ON_LINT=true`.

The test suite uses testthat edition 3 (`Config/testthat/edition: 3`). Tests are
written with `expect_lint()` and `expect_no_lint()`. `expect_no_lint(code, linter)`
asserts no lint; `expect_lint(code, message_regex, linter)` asserts a match.
You can assert lint attributes (`message`, `line_number`, `column_number`,
`type`) and pass a list of lists to assert multiple lints. The `trim_some()`
helper (in `tests/testthat/helper.R`) dedents multi-line test code.

## Contribution and PR expectations

- File or reference an issue for non-trivial changes before opening a PR.
- Reference the issue in the PR description (`Fixes #N`).
- Keep PRs focused. Split large changes into smaller, reviewable PRs.
- Add a `NEWS.md` bullet for user-facing changes, written for users.
- Match the surrounding code: idiomatic tidyverse R, the native `|>` pipe, and
  existing XPath/linter patterns.
- Add genuine technical terms to `inst/WORDLIST` rather than disabling the
  spell check.

## Things to avoid

- Editing generated files by hand: `man/`, `NAMESPACE`, and `README.md` (built
  from `README.Rmd`).
- Adding a linter without updating `inst/lintr/linters.csv` and the docs. The
  `repo-meta-tests` job will fail.
- Implementing linter logic by scanning raw text. Work from the parse-tree XML
  to avoid false positives in comments and strings.
- Adding new package dependencies without a clear reason. Declare any in
  `DESCRIPTION` (Imports/Suggests) rather than loading ad hoc.
- Reformatting unrelated files or bundling unrelated refactors into a fix.
